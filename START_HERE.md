## 1. Что решает проект

Репозиторий исследует монохромную версию задачи «никакие три точки не лежат на одной прямой» на шахматной сетке.

Для доски `n × n` выбираются точки только одного цвета, то есть одной чётности:

```text
x + y ≡ 0 (mod 2)
```

или

```text
x + y ≡ 1 (mod 2).
```

Нужно выбрать как можно больше точек так, чтобы никакие три выбранные точки не лежали на одной прямой любого наклона.

Обозначение:

```text
D_mono(n)
```

— наибольшее возможное число таких точек.

Важно: проверять нужно **все прямые**, а не только строки, столбцы и диагонали наклонов `+1` и `-1`.

## 2. Текущий подтверждённый математический статус

| Размер | Точное значение | Статус |
|---:|---:|---|
| `17×17` | `D_mono(17) = 26` | закрыт |
| `18×18` | `D_mono(18) = 27` | закрыт |
| `19×19` | `D_mono(19) = 29` | закрыт |
| `20×20` | `D_mono(20) = 30` | закрыт |
| `21×21` | `D_mono(21) = 32` | закрыт |
| `22×22` | `33 ≤ D_mono(22) ≤ 34` | активный фронтир |

Закрытым размер считается только тогда, когда есть обе части:

1. проверенная конфигурация нижней границы;
2. проверяемое доказательство верхней границы — сертификат или полный точный перебор.

Нельзя объявлять размер закрытым только потому, что решатель долго не нашёл более крупную конфигурацию.

## 3. Текущая задача: доска 22×22

В папке `22x22/` есть проверенная конфигурация из 33 точек:

```bash
python 22x22/verify_22x22.py
```

Она доказывает:

```text
D_mono(22) ≥ 33.
```

Активная цель — строго исключить конфигурацию из 34 точек.

Для чётного размера два цвета равносильны отражением

```text
(x, y) -> (21 - x, y),
```

поэтому достаточно проверить одну чётность.

При цели 34 точки:

- в каждой строке не более двух точек;
- в каждом столбце не более двух точек;
- число строк с двумя точками лежит от 12 до 17;
- число столбцов с двумя точками лежит от 12 до 17;
- транспонирование меняет эти два числа местами.

Поэтому исходное разбиение содержит 21 случай:

```text
12 ≤ row_twos ≤ column_twos ≤ 17.
```

Каждая точная модель обязана содержать ограничения для **всех максимальных прямых**, на которых лежат хотя бы три разрешённые точки.

## 4. Что произошло в двух проблемных прогонах

### Прогон 29533939978

Ссылка:


Правильная защита уже добавлена:

```python

```

Результаты этого запуска нельзя считать исследовательским перебором верхней границы.

Ссылка:

```text
https://github.com/Grisha-Pochuev/monochromatic-no-three-in-line-solver/actions/runs/29534667942
```

Этот запуск технически работал правильно.

Итог:

- проверка 33-точечной конфигурации прошла;
- 20 из 21 раздела отработали примерно по 21 000 секунд;
- все 20 сохранённых разделов получили статус `UNKNOWN`;
- ни один раздел не нашёл 34-точечную конфигурацию;
- ни один раздел не доказал `INFEASIBLE`;

- сохранено около **116,67 вычислительных часа**;

Результаты сохранены здесь:

```text

```

Главный вывод: простое разбиение только по числу насыщенных строк и столбцов слишком грубое. Повторять тот же шестичасовой запуск только с другими случайными числами — плохая стратегия.

CP-SAT не умеет буквально продолжать сохранённое состояние поиска после `UNKNOWN`. Старые JSON-файлы являются ценной статистикой и исходной точкой сравнения, но не контрольной точкой, из которой решатель продолжит то же дерево.

## 5. Усиленная точная модель

Файл:

- строки;
- столбцы;
- диагонали наклона `+1`;

Эти рисунки связываются с общими переменными точек. Остальные направления также получают полные ограничения «не более двух точек на прямой».

## 6. Как правильно понимать статусы решателя

Это критически важно.

### `INFEASIBLE`

Для точной модели это доказательство, что рассматриваемый раздел невозможен.

Если все разделы полного разбиения получили `INFEASIBLE`, верхняя граница доказана.

### `FEASIBLE` или `OPTIMAL`

Найдена допустимая конфигурация целевого размера.

Для текущей цели 34 точки это будет контрпример к предполагаемой верхней границе 33. Конфигурацию нужно отдельно проверить независимым скриптом.

### `UNKNOWN`

Решатель не успел ни найти конфигурацию, ни доказать её невозможность.

Это означает только:

```text
математический вопрос в этом разделе остаётся открытым.
```

`UNKNOWN` — не доказательство и не ошибка модели.

### `MODEL_INVALID`

Настоящая техническая ошибка модели или параметров. Такой результат нельзя использовать математически.

### `cancelled`

Задание было остановлено снаружи или упёрлось в предел рабочего процесса. Это неполный результат.

### Красный крест GitHub Actions

Красный крест сам по себе ничего не говорит о математике.

Раньше программа намеренно возвращала код `2` при `UNKNOWN`, из-за чего честный незавершённый поиск показывался как техническая ошибка. В новых процессах `UNKNOWN` должен сохраняться зелёным техническим заданием, а математическая незавершённость записывается внутри `overall.json` и `SUMMARY.md`.

## 7. Обязательный порядок перед новым тяжёлым запуском

### Шаг 1. Прочитать этот файл

В новом чате сначала открыть `START_HERE.md`. Не начинать работу по памяти старого чата.

### Шаг 2. Проверить GitHub Actions

Перед запуском посмотреть:

- нет ли уже идущего тяжёлого прогона;
- нет ли такого же прогона в очереди;
- какой SHA коммита использует текущий запуск;
- не запущен ли случайно старый рабочий процесс.

Не запускать два одинаковых тяжёлых расчёта одновременно без специальной причины.

### Шаг 3. Проверить модель коротко

До часового или шестичасового расчёта провести короткую проверку:

- скрипт запускается;
- зависимости устанавливаются;
- артефакт создаётся;
- JSON читается сборщиком;
- все случаи матрицы перечислены;
- `UNKNOWN` не превращается в техническое падение;
- `MODEL_INVALID` действительно останавливает задание.

### Шаг 4. Закреплять версии зависимостей

Не использовать без необходимости:

```bash
pip install --upgrade ortools
```

Предпочтительно:

```bash
pip install ortools==9.15.6755
```

Это защищает воспроизводимость от неожиданного изменения поведения новой версии.

### Шаг 5. Проверять случайные числа

Всегда приводить `random_seed` к допустимому диапазону:

```python
seed % 2_147_483_647
```

В JSON сохранять оба значения:

- `requested_seed`;
- `effective_seed`.

### Шаг 6. Оставлять запас по времени

Внутренний лимит решателя должен быть меньше лимита задания GitHub Actions.

Пример для часового задания:

```text
timeout-minutes: 60
solver seconds: примерно 3500
```

Не отдавать решателю все 3600 секунд, иначе не останется времени записать JSON и загрузить артефакт.

Если в одном задании идут два случая, их суммарное время вместе с установкой зависимостей и загрузкой файлов должно помещаться в лимит.

### Шаг 7. Учитывать реальные машины GitHub

Обычная рабочая схема проекта:

```text
20 параллельных заданий
4 потока CP-SAT на задание
примерно 14–16 ГБ памяти на задание
```

Процессор может отличаться между заданиями. Нельзя делать вывод о качестве метода только по скорости одного раздела.

### Шаг 8. Артефакты загружать всегда

Шаг загрузки должен иметь:

```yaml
if: always()
```

JSON нужно записывать до завершения программы. Даже `UNKNOWN` полезен как исследовательский результат.

### Шаг 9. Отделять технический успех от математического

Отдельное вычислительное задание должно быть зелёным, если:

- программа корректно запустилась;
- модель была допустимой;
- результат записан;
- артефакт сохранён.

Общий сборщик должен отдельно записывать математический вывод:

```text
PROVED
FOUND_COUNTEREXAMPLE
INCOMPLETE
TECHNICAL_FAILURE
```

Не следует делать всю доску красной только потому, что доказательство осталось открытым.

### Шаг 10. Не закрывать размер преждевременно

Размер 22×22 можно объявить закрытым со значением 33 только если:

- 33-точечная конфигурация независимо проверена;
- все части полного разбиения цели 34 получили `INFEASIBLE`;
- итоговая сводка содержит все ожидаемые записи;
- отсутствуют `UNKNOWN`, пропущенные и отменённые разделы.

## 8. Как сохранять каждый значимый прогон

Для каждого важного запуска создавать отдельную папку:

```text
22x22/runs/YYYY-MM-DD-run-RUN_ID/
```

Минимальный набор:

```text

число ожидаемых и полученных разделов
статусы INFEASIBLE / UNKNOWN / FEASIBLE

```

По возможности сохранять исходные JSON каждого раздела.

После разбора прогона:

1. сделать коммит результатов;
2. обновить README соответствующего размера;
3. обновить корневой README только при изменении подтверждённого статуса;
4. обновить `START_HERE.md`;
5. указать, какой файл является точкой продолжения.

Нельзя подменять результаты старого запуска новым рабочим процессом. У каждого запуска должны сохраняться его собственные модель, параметры и SHA.


### Текущая лучшая линия атаки

1. Усиленная модель точных рисунков строк, столбцов и двух диагональных семейств.

Каждое новое разбиение должно быть доказанно полным: все возможные значения параметра должны быть покрыты.

## 10. Общая исследовательская линия после 22×22

Не следует превращать проект только в последовательность всё более дорогих переборов.

Параллельно нужно развивать три направления:

### Нижние конструкции

Искать хорошие конфигурации с помощью:

- CP-SAT;
- стохастического SAT;
- локального поиска;
- геометрического поиска с сохранением допустимых ходов;
- симметричных и почти симметричных шаблонов.

### Верхние границы

Использовать:

- четырёхнаправленные линейные и целочисленные сертификаты;
- точные CP-SAT-разбиения;
- дополнительные семейства наклонов;
- маленькие независимо проверяемые сертификаты, когда это возможно.

### Закономерности

Сохранять для лучших конфигураций:

- распределение точек по строкам;
- распределение по столбцам;
- распределение по диагоналям `+1` и `-1`;
- наиболее опасные другие наклоны;
- симметрии;
- различия между чётными и нечётными размерами.

Наблюдаемая формула `ceil(3n/2)` хорошо описывает ряд малых размеров, но профессор Томас Прелльберг нашёл более крупные симметричные конфигурации для некоторых больших `n`. Поэтому её нельзя считать общей верхней границей без отдельного доказательства.
