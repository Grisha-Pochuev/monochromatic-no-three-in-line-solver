## 1. Что решает проект

Для доски `n × n` выбираются точки только одного шахматного цвета:

```text
x + y ≡ 0 (mod 2)
```

или

```text
x + y ≡ 1 (mod 2).
```

Нужно выбрать как можно больше точек так, чтобы никакие три выбранные точки не лежали на одной прямой любого наклона.

Обозначение:

```text
D_mono(n)
```

— наибольшее возможное число таких точек.

Важно: проверять нужно **все прямые**, а не только строки, столбцы и диагонали наклонов `+1` и `-1`.

## 2. Обязательная структура репозитория

### Для каждого размера доски — отдельная папка в корне

Каждый новый размер доски должен иметь собственную папку непосредственно в корне репозитория.

Формат названия строго такой:

```text
17x17/
18x18/
19x19/
20x20/
21x21/
22x22/
23x23/
...
```

Нельзя использовать названия вроде:

```text
N17/
n22/
board-23/
```

Нельзя складывать файлы разных размеров в одну общую папку.

Когда начинается работа над новым размером, сначала создаётся его корневая папка, например:

```text
23x23/
```

В папке конкретного размера должны находиться относящиеся к нему:

- `README.md` с человеческим описанием статуса;
- лучшие найденные конфигурации;
- независимые проверяющие скрипты;
- программы поиска и доказательства верхней границы;
- сертификаты и итоговые JSON;
- заметки о текущем фронтире и следующем шаге;
- папка `runs/` с архивами значимых запусков.

Рекомендуемая схема:

```text
22x22/
├── README.md
├── verify_22x22.py
├── config_33.json
├── prove_upper_22x22_pattern_model.py
├── notes/
└── runs/
    ├── 2026-07-17-run-29534667942/
    └── 2026-07-17-run-29561750178/
```

Общие файлы, относящиеся ко всему проекту, остаются в корне:

```text
README.md
START_HERE.md
LICENSE
.github/
```

Workflow-файлы GitHub Actions хранятся в:

```text
.github/workflows/
```

Но результаты конкретного workflow после завершения должны архивироваться внутри папки соответствующего размера.

## 4. Текущая задача: 22×22

В папке `22x22/` есть проверенная конфигурация из 33 точек:

```bash
python 22x22/verify_22x22.py
```

Она доказывает:

```text
D_mono(22) ≥ 33.
```

Активная цель — строго исключить конфигурацию из 34 точек либо найти такую конфигурацию.

Для чётного размера два цвета равносильны отражением:

```text
(x, y) -> (21 - x, y).
```

При цели 34 точки:

- в каждой строке не более двух точек;
- в каждом столбце не более двух точек;
- число строк с двумя точками лежит от 12 до 17;
- число столбцов с двумя точками лежит от 12 до 17;
- транспонирование меняет эти два числа местами.

Исходное полное разбиение содержит 21 случай:

```text
12 ≤ row_twos ≤ column_twos ≤ 17.
```

Каждая точная модель обязана содержать ограничения для **всех максимальных прямых**, на которых лежат хотя бы три разрешённые точки.

## 7. Как понимать статусы

### `INFEASIBLE`

Рассматриваемый раздел строго невозможен. Если все части полного разбиения получили `INFEASIBLE`, верхняя граница доказана.

### `FEASIBLE` или `OPTIMAL`

Найдена конфигурация целевого размера. Её нужно отдельно проверить независимым скриптом.

### `UNKNOWN`

Решатель не успел ни найти конфигурацию, ни доказать невозможность.

`UNKNOWN` — не доказательство и не техническая ошибка.

### `MODEL_INVALID`

Настоящая ошибка модели или параметров. Такой результат математически непригоден.

### `cancelled`

Задание остановлено снаружи или упёрлось в общий предел. Это неполный результат.

### Красный крест GitHub Actions

Красный крест сам по себе ничего не говорит о математическом результате.

`UNKNOWN` не должен превращаться в техническое падение. Технически корректное задание должно сохранить JSON и артефакт, а математическая незавершённость записывается внутри сводки.

## 8. Обязательный порядок перед тяжёлым запуском

### Шаг 1. Прочитать этот файл

В новом чате сначала открыть `START_HERE.md` один раз.

### Шаг 2. Проверить очередь Actions

Проверить:

- нет ли уже идущего тяжёлого прогона;
- нет ли такого же прогона в очереди;
- какой SHA коммита использует запуск;
- не запущен ли старый workflow.

Не запускать два одинаковых тяжёлых расчёта одновременно без специальной причины.

### Шаг 3. Провести короткую проверку

Перед часовым или шестичасовым запуском проверить за несколько минут:

- скрипт запускается;
- зависимости устанавливаются;
- все части разбиения перечислены;
- JSON создаётся и читается сборщиком;
- артефакт загружается;
- `UNKNOWN` не вызывает красный технический статус;
- `MODEL_INVALID` останавливает задание.

### Шаг 4. Закреплять версии

Предпочтительно:

```bash
pip install ortools==9.15.6755
```

Не обновлять OR-Tools без причины непосредственно перед большим запуском.

### Шаг 5. Проверять seed

```python
effective_seed = requested_seed % 2_147_483_647
```

В JSON сохранять оба значения:

- `requested_seed`;
- `effective_seed`.

### Шаг 6. Оставлять запас времени

Для задания с лимитом 60 минут решателю давать примерно 3500 секунд, а не все 3600. Остаток нужен для записи JSON и загрузки артефакта.

### Шаг 7. Учитывать машины GitHub

Обычная схема проекта:

```text
20 параллельных заданий
4 потока CP-SAT на задание
примерно 14–16 ГБ памяти на задание
```

Процессоры разных заданий могут отличаться. Скорость одного раздела нельзя считать общей оценкой модели.

### Шаг 8. Всегда сохранять артефакты

Шаг загрузки должен использовать:

```yaml
if: always()
```

JSON нужно записывать до выхода программы. Даже `UNKNOWN` полезен как исследовательский результат.

### Шаг 9. Разделять технический и математический итог

Сводка должна различать:

```text
PROVED
FOUND_COUNTEREXAMPLE
INCOMPLETE
TECHNICAL_FAILURE
```

### Шаг 10. Не закрывать размер преждевременно

Размер 22×22 можно закрыть со значением 33 только если:

- 33-точечная конфигурация независимо проверена;
- все части полного разбиения цели 34 получили `INFEASIBLE`;
- итоговая сводка содержит все ожидаемые части;
- нет `UNKNOWN`, пропущенных или отменённых частей.

## 9. Как сохранять каждый значимый прогон

Для каждого важного запуска создавать отдельную папку внутри папки его доски:

```text
NxN/runs/YYYY-MM-DD-run-RUN_ID/
```

Например:

```text

```

Минимальный набор:

```text

исходные JSON разделов

число ожидаемых и полученных разделов

```

После разбора прогона:

1. сделать коммит результатов;
2. обновить `README.md` соответствующего размера;
3. обновить корневой README только при изменении подтверждённого статуса;
4. обновить `START_HERE.md`;
5. записать точную точку продолжения.

Нельзя подменять результаты старого запуска новым workflow. У каждого запуска должны сохраняться собственные модель, параметры и SHA.

## 10. Исследовательская линия по 22×22

Уже известно:

- 33 точки существуют и проверены;
- долгий поиск не нашёл 34 точки;
- это ещё не доказывает невозможность 34 точек;

Текущая линия атаки:

1. сравнить усиленную модель со старой на всех 21 разделах;

Перспективный параметр:

```text
число диагоналей наклона +1, содержащих ровно две выбранные точки
```

Также можно исследовать:

- такой же параметр для диагоналей `-1`;
- граничные профили;
- симметричные и несимметричные классы;
- точные шаблоны насыщенных диагоналей;
- дополнительное разбиение только тяжёлых разделов.

Каждое новое разбиение должно быть доказанно полным: все допустимые значения параметра обязаны быть покрыты.

Наблюдаемая формула `ceil(3n/2)` хорошо описывает несколько малых размеров, но не является доказанной общей верхней границей.

## 12. Правила работы в веб-чате

Пользователю объяснять технические результаты простым русским языком: сначала человеческий смысл, затем подробности.

При работе:

- не начинать исследование заново;
- не повторять уже проверенные вопросы;
- не заявлять о закрытии без строгого подтверждения;
- после запуска давать прямую ссылку;
- после завершения кратко писать: что доказано, что не доказано и что делать дальше;
- тяжёлые расчёты выполнять через GitHub Actions;
- локально делать только короткие проверки;
- при ошибке сначала собирать логи и артефакты;
- после значимого шага коммитить результаты и обновлять этот файл.
