## 1. Задача

На доске `n × n` выбираются точки одного шахматного цвета. Нужно максимизировать их число при условии, что никакие три выбранные точки не лежат на одной прямой любого наклона. Максимум обозначается `D_mono(n)`.

Проверять нужно **все прямые**, а не только строки, столбцы и диагонали `±1`.

---

## 4. Активная задача: 22×22

В `22x22/config_33.json` есть проверенная конфигурация из 33 точек:

```bash
python 22x22/verify_22x22.py
```

Она доказывает `D_mono(22) ≥ 33`. Нужно либо найти 34 точки, либо строго доказать их невозможность.

Для чётной доски две чётности равносильны отражением `(x,y) -> (21-x,y)`, поэтому достаточно одной чётности. Каждая точная модель обязана содержать ограничения для всех максимальных прямых, содержащих не менее трёх разрешённых точек.

## 7. Новое pair/profile-разбиение

Не повторять 88 оставшихся пар целиком.

Для любой 34-точечной конфигурации:

```text
diag_plus_twos  ∈ {13,14,15,16,17}
diag_minus_twos ∈ {12,13,14,15,16,17}
```

Поэтому каждая из 88 пар делится ровно на 30 детей. Полное разбиение содержит:

```text
88 × 5 × 6 = 2640 детей.
```

После фиксации пары из общего запаса сертификата вычитается excess двух выбранных точек. Остаточный запас для 88 пар лежит между 53 и 112. Это даёт точные усиления:

1. каждая сертификатная линия с весом больше остатка обязана быть насыщена;
2. остальные выбранные точки подчиняются уменьшенному excess-бюджету;
3. добавляются пороговые ограничения на число точек с большим excess;
4. явные состояния `0/1/2` точек на строках, столбцах и диагоналях подчиняются точному взвешенному бюджету дефекта линий;
5. транспонирование безопасно разрывается лексикографически, потому что оба числовых профиля при транспонировании сохраняются.

Точный динамический расчёт минимального дефекта отбрасывает больше половины 2640 детей ещё без CP-SAT. Локально осталось около 1130 детей, требующих настоящего поиска.

Файлы:

- `22x22/prove_upper_22x22_profile_child.py`;
- `22x22/run_n22_profile_smoke_shard.py`;
- `22x22/collect_n22_profile_smoke.py`;
- `22x22/PROFILE_SPLIT_RESEARCH_2026-07-20.md`;

## 8. Результаты локальных проверок новой модели

Это исследовательские тесты времени; публичный Actions-run нужен для полного воспроизводимого протокола.

- За 1,5 секунды на каждый из пяти `diag_plus_twos`-случаев по всем 88 парам строго закрыты 145 из 440 детей.
- На девяти парах с минимальным остаточным запасом пятисекундный тест закрыл 41 из 81 допустимого сочетания `(plus_twos,minus_twos)`.
- Трудный ребёнок `(pair=2, plus=16, minus=15)` закрылся примерно за девять секунд при четырёх работниках CP-SAT.

Точные рисунки диагоналей могут усилить следующий этап, но при их фиксации нельзя слепо сохранять transpose-lex: транспонирование переводит рисунок `k` в рисунок `-k`. Точные рисунки надо либо канонизировать по орбитам транспонирования, либо решать без этого отсечения.

## 9. Подготовленный следующий прогон

Workflow:

```text
.github/workflows/n22-profile-children-smoke.yml
```

Он покрывает все 2640 детей ровно один раз на 20 шардах:

- арифметически невозможные дети закрываются немедленно;
- остальные получают 20 секунд и 4 работника CP-SAT;
- ожидаемое время по стене — примерно 15–25 минут;
- сборщик проверяет полноту, найденные решения и число полностью закрытых пар.

В лучшем случае этот прогон закроет все 88 пар и докажет `D_mono(22)=33`. Если останутся `UNKNOWN`, он даст точный малый список детей для следующего разбиения по каноническим рисункам диагоналей.

## 10. Правила тяжёлых запусков

1. Сначала прочитать этот файл.
2. Проверить очередь Actions, SHA и отсутствие дублирующего запуска.
3. Перед длинным запуском проверять движок, разбиение, JSON, сборщик и загрузку артефактов.
4. Использовать `ortools==9.15.6755` во всех заданиях, включая сборщик.
5. Seed приводить к знаковому 32-битному диапазону и сохранять оба значения.
6. Использовать до 20 параллельных заданий и 4 потока CP-SAT на задание.
7. Всегда сохранять артефакты через `if: always()`.
8. Отличать математический `UNKNOWN` от технической ошибки.
9. Не закрывать доску, пока остаётся хотя бы один `UNKNOWN`, пропуск или отменённый случай.
10. После завершения сохранять результаты в папке размера, обновлять README и этот файл.
